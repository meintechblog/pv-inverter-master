# v4.0 Milestone Audit: Multi-Source Virtual Inverter

**Scope:** Phases 21–24 (8 plans, 28 requirements)
**Auditor:** Integration checker
**Date:** 2026-03-21

---

## Executive Summary

The v4.0 milestone is **substantively complete**. All 28 requirements are implemented and verified at the code level, all four phases delivered their planned artifacts, and all critical cross-phase wiring paths are connected. However, three categories of gap prevent a clean `passed` status:

1. **9 broken tests** that were not repaired after Phase 24 restructured the frontend (8 assert the old navigation contract; 1 is a pre-existing SF=0 migration failure).
2. **Two legacy API endpoints** (`/api/registers`, `/api/dashboard`) still serve first-device-only data while the frontend's Registers tab continues to call `/api/registers` — incorrect for multi-device deployments.
3. **One fragile internal coupling**: the webapp accesses `PowerLimitDistributor` via private-attribute reflection (`getattr(registry, "_distributor", None)`) rather than a public accessor, making the wiring invisible to type-checkers and fragile to refactoring.

No requirement is missing from the codebase. No E2E user flow has a hard break; however, one flow (view registers of a specific inverter) delivers incorrect data when more than one inverter is configured.

**Fragile coupling identified:** `webapp.py` and `ws_handler` reach the distributor via three-level private-attribute reflection:

```python
distributor = getattr(getattr(app_ctx, "device_registry", None), "_distributor", None)
device_limits = distributor.get_device_limits() if distributor is not None else {}
```

`_distributor` is a private attribute set by `__main__.py` after `run_modbus_server` returns the slave context. The webapp has no direct reference to the distributor; it must navigate `app_ctx -> device_registry -> _distributor`. This path works at runtime but is invisible to type-checkers, fails silently when the attribute is absent (returns empty dict), and breaks if `_distributor` is renamed. A `get_distributor()` method on `DeviceRegistry` or a direct `app_ctx.distributor` field would eliminate this.

### Missing or Degraded Connections

#### 1. Register Viewer: Multi-Device Data Mismatch

**Severity:** Degraded (incorrect data, not a crash)

The frontend's register viewer calls `/api/registers` for all inverter devices:

```
src/venus_os_fronius_proxy/static/app.js lines 630, 649:
    fetch('/api/registers')
```

The `/api/registers` endpoint in `webapp.py` (line 439) explicitly notes:

```python
# TODO Phase 24: per-device register viewer
first_dev = next(iter(app_ctx.devices.values()), None)
```

In a multi-inverter deployment, clicking the Registers tab on any inverter other than whichever happens to be `next(iter(...))` returns the wrong device's registers. The endpoint was not upgraded during Phase 24 — the verification report flags it as an "Info" anti-pattern and notes that `/api/devices/{id}/snapshot` is the "correct" path, but the register viewer JS was not updated to use it.

**Fix required:** Either update `renderInverterRegisters` in `app.js` to fetch from `/api/devices/{id}/snapshot` and extract the register data, or add `/api/devices/{id}/registers` in `webapp.py`.

#### 2. Legacy `/api/dashboard` Still First-Device-Only

**Severity:** Low (the endpoint exists and works; the device-centric dashboard pages use `device_snapshot` not this endpoint)

`webapp.py:496-498`:
```python
# TODO Phase 24: aggregated virtual dashboard
first_dev = next(iter(app_ctx.devices.values()), None)
```

The frontend's device-centric dashboard pages (`renderInverterDashboard`, `renderVirtualPVPage`) do not call `/api/dashboard`; they use `/api/devices/{id}/snapshot` or WS `device_snapshot` events. The legacy endpoint is called only by the old `handleSnapshot` handler (triggered by WS `{"type": "snapshot"}` on line 293 of `app.js`), which fires when the ws_handler sends the first-device snapshot at connection time (webapp.py:550). This path is a backward-compatibility concern; no current UI renders from it actively.

## Verdict

**Status: `gaps_found`**

The milestone infrastructure is complete and the primary architecture goals are achieved: N-device polling, virtual inverter aggregation, priority-based power limiting, and a device-centric frontend all work end-to-end. The gaps are:

- 1 degraded flow (register viewer shows first device's data for all devices)
- 3–10 broken tests from deferred SF=0 migration and frontend restructure
- 1 fragile private-attribute access pattern for distributor discovery

None of these are architecture failures. All 28 requirements have working implementations. The milestone can be considered **functionally complete** pending the test suite repairs and register viewer fix.
